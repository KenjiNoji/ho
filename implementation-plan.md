# Pomodoro Timer App Implementation Plan

## 目的

このドキュメントは、ポモドーロタイマー Web アプリケーションを段階的に実装するための計画をまとめたものである。

添付の UI モック、[architecture.md](architecture.md)、[features.md](features.md) を前提に、機能を無理なく積み上げられる順序で整理する。

計画の方針は次の通りである。

- 各段階で 1 つのユーザー価値を成立させる
- 各段階で 1 つから 2 つの技術的関心に絞る
- 段階の終わりで動作確認またはテストで完了判定できるようにする
- UI とロジックを分離し、後から機能を追加しやすくする

## 実装粒度の考え方

実装粒度は、大きすぎず小さすぎない単位にする。

- 1 段階の終わりで、ブラウザ操作またはテストで成果を確認できること
- 1 段階で変更する関心事は 1 つか 2 つまでに絞ること
- タイマーロジックは UI に接続する前に単独で固めること
- 保存機能や API は基本操作が安定してから追加すること

## 段階的実装計画

### 段階 1. 開発土台の構築

この段階では、アプリケーションを起動してブラウザで画面を表示できる状態を作る。

#### 実装対象

- Flask の app factory 構成
- トップページ用 route
- templates と static の基本構成
- HTML の基本骨格
- CSS と JavaScript の読み込み設定
- JavaScript の初期化エントリポイント

#### 完了条件

- Flask サーバーが起動できる
- ブラウザでトップページが表示できる
- ポモドーロアプリの空の画面骨格が見える

#### この段階の狙い

- 実装を載せる土台を先に安定させる
- 以降の UI 実装とロジック実装の受け皿を作る

### 段階 2. UI モックの静的再現

この段階では、添付モックをもとに見た目を HTML と CSS で再現する。

#### 実装対象

- タイマー表示領域
- モード切替タブまたはボタン
- 開始、一時停止、再開、リセットなどの操作ボタン
- 設定パネルのレイアウト
- 完了回数表示の表示枠
- デスクトップとモバイルの基本レスポンシブ対応

#### 完了条件

- 主要 UI が静的に表示される
- デスクトップとモバイルでレイアウトが崩れない
- モックに近い見た目の方向性が確認できる

#### この段階の狙い

- 見た目と情報設計を先に固める
- 後続のロジック接続時に DOM 構造を大きく変えなくて済むようにする

### 段階 3. タイマーエンジンの単体実装

この段階では、UI と切り離した純粋ロジックとしてタイマーエンジンを実装する。

#### 実装対象

- state モデル
- mode 管理
- 開始
- 一時停止
- 再開
- リセット
- 終了予定時刻ベースの残り時間計算
- focus 完了後の break 遷移
- Clock 抽象

#### 完了条件

- DOM に依存せず状態遷移が成立する
- 基本的なユニットテストが通る
- 残り時間計算とモード遷移が期待通りに動く

#### この段階の狙い

- もっとも壊れやすいタイマーロジックを先に固める
- UI の不具合とロジックの不具合を切り分けやすくする

### 段階 4. タイマー基本操作の UI 接続

この段階では、タイマーエンジンを画面へ接続して基本操作を成立させる。

#### 実装対象

- renderer
- イベント登録
- 開始処理の UI 接続
- 一時停止処理の UI 接続
- 再開処理の UI 接続
- リセット処理の UI 接続
- 現在モード表示
- 残り時間のリアルタイム更新

#### 完了条件

- ブラウザ上で開始、一時停止、再開、リセットができる
- UI 表示と内部 state が一致する
- タイマーが正しく進行して見える

#### この段階の狙い

- 最初の実用的なユーザー価値を提供する
- state と view の接続方式を固める

### 段階 5. モード切替とポモドーロサイクル管理

この段階でポモドーロアプリとしての基本挙動を完成させる。

#### 実装対象

- 作業モードの手動切替
- 短い休憩モードの手動切替
- 長い休憩モードの手動切替
- focus 完了回数の管理
- 規定回数ごとの long break 判定
- 完了回数表示

#### 完了条件

- 手動で各モードに切り替えられる
- 作業完了を繰り返すと short break と long break が正しく切り替わる
- 完了回数表示が正しく更新される

#### この段階の狙い

- ポモドーロ固有のサイクルを成立させる
- ただのカウントダウンタイマーから脱却する

### 段階 6. 設定変更とバリデーション

この段階では、ユーザーが自分に合った設定へ変更できるようにする。

#### 実装対象

- 作業時間の設定変更
- 短い休憩時間の設定変更
- 長い休憩時間の設定変更
- long break 間隔の設定変更
- 入力値バリデーション
- エラー表示

#### 完了条件

- 妥当な設定値だけが保存または反映される
- 不正な値は UI 上で明確に拒否される
- 設定変更後にタイマーへ正しく反映される

#### この段階の狙い

- アプリを個人の使い方へ適応させる
- 入力不正による state 崩壊を防ぐ

### 段階 7. 保存と復元

この段階では、ページ再読み込み後も継続利用できる状態にする。

#### 実装対象

- Storage 抽象
- localStorage 実装
- 設定値の保存
- 現在モードの保存
- 必要に応じた残り時間または終了予定時刻の保存
- 読み込み時の state 復元

#### 完了条件

- ページ再読み込み後に設定値を復元できる
- 必要なアプリ状態を復元できる
- 保存と復元でタイマー挙動が破綻しない

#### この段階の狙い

- 実運用しやすいアプリにする
- 将来のサーバー保存への差し替えを見据えて抽象を導入する

### 段階 8. 通知と自動開始設定

この段階では、ユーザー体験を完成に近づける。

#### 実装対象

- 作業終了時の通知
- 休憩終了時の通知
- 画面通知
- 音通知
- 可能ならブラウザ通知
- 休憩自動開始設定
- 作業自動開始設定

#### 完了条件

- セッション終了時にユーザーが確実に気づける
- 自動開始設定が期待通りに動作する
- 通知機能が基本操作と干渉しない

#### この段階の狙い

- 日常利用に耐える体験へ仕上げる
- 毎回操作しなくても流れが継続できるようにする

### 段階 9. Flask API の導入準備

この段階では、将来のサーバー保存や統計機能に備えてサーバー側の構造を整える。

#### 実装対象

- route と service の分離
- 設定取得 API の雛形
- 設定保存 API の雛形
- 統計取得 API の雛形
- フロントエンド側の API 利用切替を見据えた構成整理

#### 完了条件

- Flask 側で API を拡張しやすい構成になっている
- フロントエンドの保存処理が将来 API 実装に切り替えやすい
- HTTP 層と業務ロジック層が分離されている

#### この段階の狙い

- 初版を壊さずに拡張準備を進める
- 保存や統計の責務を明確にする

### 段階 10. 統計と履歴の拡張

この段階は初版公開後でもよい拡張フェーズとする。

#### 実装対象

- セッション完了履歴の保存
- 当日の完了回数表示
- 総作業時間表示
- 日別集計
- 必要に応じた SQLite 永続化

#### 完了条件

- 作業実績をユーザーが振り返れる
- 履歴と統計が継続的に保存される
- 将来の分析機能に繋げられるデータ構造になっている

#### この段階の狙い

- 初版以降の付加価値を提供する
- 単なるタイマーから継続利用したくなるアプリへ広げる

## マイルストーン

段階をまとめると、以下のマイルストーンで管理しやすい。

### マイルストーン 1

- 段階 1 と段階 2
- 画面が出る
- 見た目がモックに近い

### マイルストーン 2

- 段階 3 と段階 4
- 開始、一時停止、再開、リセットが動く

### マイルストーン 3

- 段階 5
- ポモドーロサイクルが正しく回る

### マイルストーン 4

- 段階 6 と段階 7
- 設定変更、保存、復元ができる

### マイルストーン 5

- 段階 8
- 通知と自動開始まで含めて初版完成

### マイルストーン 6

- 段階 9 と段階 10
- 統計とサーバー保存を加えた拡張版完成

## 実装順の要点

- 先に土台と静的 UI を作る
- 次にタイマーエンジンを UI から独立して作る
- その後で UI に接続する
- 基本操作が安定してから設定、保存、通知を追加する
- API と統計は初版完成後または拡張フェーズで扱う

## 推奨する進め方

現実的には、以下のようなまとまりで進めると安定しやすい。

1. 1 回目の実装で、土台と静的 UI を作る
2. 2 回目の実装で、タイマーエンジンとユニットテストを作る
3. 3 回目の実装で、UI とロジックを接続して基本操作を完成させる
4. 4 回目の実装で、モード切替、設定、保存を追加する
5. 5 回目の実装で、通知と自動開始を追加して初版を完成させる
6. その後、必要性に応じて API、統計、履歴保存を拡張する

## 結論

このアプリケーションは、一度に全機能を作るのではなく、土台、UI、ロジック、接続、設定、保存、通知、拡張の順で段階的に積み上げるのが最も安全である。

特に重要なのは、タイマーエンジンを UI から分離して早い段階で固めることと、各段階の終わりに必ず動作確認できる状態を作ることである。

この計画に沿って進めることで、破綻しにくく、テストしやすく、拡張可能なポモドーロタイマーアプリケーションを構築できる。
